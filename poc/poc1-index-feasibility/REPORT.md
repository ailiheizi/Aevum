# PoC-1:包索引依赖元数据可机器生成性 — 实测报告

> 日期:2026-06-08 · 状态:已完成
> 目的:用真实数据证伪/证实评审报告 C1(头号风险)——"Aevum 的包索引(provides/requires)能不能从二进制自动生成,还是必须像 nixpkgs 那样靠十年人工?"
> 数据源:Debian stable `main/binary-amd64` 全量 `Packages.gz`(68,755 包)+ 19 个真实 .deb 抽样解 ELF

---

## 0. 为什么做这个 PoC

评审把 **C1(生态/索引来源缺失)** 列为唯一的 CRITICAL、整份设计的存亡级前提:Aevum 的求解器、世代、AI 维护者全部建立在"存在一个已标注 `provides`/`requires` 的可信包索引"之上,而文档对这个索引的内容从哪来只字未提。

与其继续补文档,不如先回答一个能被数据证伪的问题:

> **依赖元数据,机器能自动生成多少?**

如果答案是"95% 自动",C1 是几个月工程;如果是"一半要人工语义",C1 就是又一个 nixpkgs 级人力黑洞,定位需要调整。

---

## 1. 方法

**Step A(静态,全量)**:解析 Debian 人工维护的 `Depends` 字段,把 308,807 条依赖原子分类:
- `lib_soname`:形如 `libssl3`、`libc6`,无版本约束 → ELF 的 `DT_NEEDED` 理论上可全自动生成。
- `lib_versioned`:库依赖 + 版本约束 `(>= 3.0)` → 库名可自动,**选哪个版本是语义决策**。
- `versioned_nonlib` / `non_lib` / `virtual_or_alt`:非库依赖(daemon、数据包、配置)、虚包、alternatives(`a | b`)→ **ELF 永远看不到,纯人工语义**。

**Step B(抽样,实证)**:下载 19 个真实包,纯 Python 解每个 ELF 的 `DT_NEEDED`,比对"自动抽出的 so 依赖"与"人工声明的依赖",测量人工声明里有多大比例是 ELF 根本不可见的。

> 工具链:纯 Python 解析 `.deb`(ar 归档)、ELF `.dynamic`/`DT_NEEDED`,不依赖 readelf/ar;Windows 上即可运行。脚本:`analyze_depends.py`、`verify_elf_coverage.py`。

---

## 2. 结果

### 2.1 Step A — 全量 308,807 条依赖的分类(Debian stable)

| 类别 | 条数 | 占比 | 能否机器生成 |
|---|---|---|---|
| `lib_soname`(纯库,无版本) | 35,871 | **11.6%** | ✅ ELF 全自动 |
| `lib_versioned`(库 + 版本约束) | 128,214 | **41.5%** | ⚠️ 库名自动,版本需语义 |
| `non_lib`(非库程序/数据) | 92,696 | 30.0% | ❌ 人工语义 |
| `versioned_nonlib` | 45,562 | 14.8% | ❌ 人工语义 |
| `virtual_or_alt`(虚包/alternatives) | 6,464 | 2.1% | ❌ 人工语义 |

**三档汇总**:

- 🟢 **纯自动可生成:11.6%**
- 🟡 **半自动(名字自动 + 版本需判定):41.5%**
- 🔴 **纯人工语义:46.9%**

**包级别**:60,333 个有依赖的包里,只有 **22.3%** 的包"全部依赖都是纯 lib soname"(理论上可全自动建条目);**8.9%** 的包用到了 alternatives(`a | b`,机器无从选择)。

### 2.2 Step B — 19 包抽样,ELF 实测

- 样本共 76 条人工声明依赖:库依赖 48 条,非库依赖 28 条。
- **36.8% 的人工声明依赖是 ELF 根本看不到的**(daemon、数据包、配置、运行时 dlopen)。

典型对照(暴露问题的最清楚):

| 包 | ELF 文件 | ELF 抽出 soname | 人工声明非库依赖 | 说明 |
|---|---|---|---|---|
| `nginx-core` | **0** | **0** | **14** | 几乎全是配置/模块/用户级语义依赖,ELF 给不出一条 |
| `bash` | 2 | 2 | 2 | 依赖 `base-files` 等,ELF 不可见 |
| `git` | 13 | 5 | 5 | perl、运行时工具依赖,ELF 只见 5 个 so |
| `imagemagick` | 0 | 0 | 1 | 元包,纯靠声明 |
| `ffmpeg` | 4 | 12 | 0 | **理想情况**:全是库依赖,ELF 几乎全覆盖 |
| `zlib1g` | 1 | 1 | 0 | 理想情况 |

→ 纯库类工具(ffmpeg、zlib、ripgrep)ELF 覆盖很好;但凡涉及服务/解释器/数据(nginx、bash、git),ELF 大面积失明。

---

## 3. 结论:C1 是真实的,但可被"分层 bootstrap"化解

### 3.1 数据裁决

评审的 C1 **成立且未被夸大**。具体到数字:

- "so 级可以、语义不行"的论断被证实:**纯自动只有 11.6%,近半(46.9%)是 ELF 永远不可见的人工语义**。
- 自己从零靠 ELF 生成一个完整可用索引 = **不可行**。一个 nginx 的 14 条语义依赖、git 的 5 条、bash 的 2 条,机器拿不到,且这些恰恰是"装完能不能跑"的关键。
- **结论:Aevum 不可能靠自动分析绕开 nixpkgs/Debian 那样的人工策展生态。这是定位级的现实。**

### 3.2 但有一条务实出路(数据也指向了它)

不是"自动生成 vs 十年人工"的二选一。真实分布给出第三条路:

```
索引 bootstrap 分层策略：
  1. 直接复用现有发行版的人工元数据（Debian/Arch/nixpkgs 的 Depends/provides）
     → 那 46.9% 的人工语义,别自己造,继承
  2. ELF 自动抽取做"校验与补全",不做"从零生成"
     → 用 11.6% 纯自动 + 41.5% 半自动,去交叉验证/修正继承来的元数据,抓漏网的 so 依赖
  3. AI 维护者的真正发力点 = 把现有发行版元数据"翻译"进 Aevum 的能力模型 + 处理冲突
     → 而非"理解二进制生成依赖"(那只有 11.6% 能成)
```

这把 C1 从"存亡级"降为"重大但可工程化":**Aevum 站在已有生态的肩膀上,用自动分析做增强层**,而不是平地起高楼。

### 3.3 对设计文档的直接影响

1. **必须新增一篇"包索引与供给模型"设计**,明确:索引内容来自复用现有发行版元数据(继承策展)+ ELF 自动校验补全 + AI 翻译进能力模型。删掉"从零靠 AI/ELF 生成"的暗示。
2. `store.toml` 里 `origin = "aevum-index:..."` 那个被命名却未定义的组件,要正式定义为"上游发行版元数据的导入 + 再签名"管线。
3. 痛点文 §7 "生态需要时间积累"一句话,要升级为正式的冷启动战略章节,并诚实写明:**Aevum 的可复现/原子是真创新,但包生态是站在 Debian/nixpkgs 肩上,不是另起炉灶**。
4. 这也反向印证了评审 H5:既然 46.9% 靠人工语义、AI 的角色是"翻译现有元数据",那 AI 模型形态(本地/云、版本锁定)更需要明确——因为 AI 在索引构建链路上是关键环节。

## 4. 复现方式

```bash
cd poc/poc1-index-feasibility
# Step A：全量静态分析
curl -s -o data/Packages.gz http://deb.debian.org/debian/dists/stable/main/binary-amd64/Packages.gz
python analyze_depends.py        # → data/step_a_result.json
# Step B：抽样 ELF 实证
python verify_elf_coverage.py    # → data/step_b_result.json
```

原始结果 JSON 在 `data/`。脚本纯 Python,无外部依赖(zstd CLI 已在 PATH 即可)。

---

## 5. 一句话总结

> **依赖元数据只有 11.6% 能从二进制纯自动生成,近半是机器永远看不见的人工语义。
> 所以 Aevum 不能自己造生态,但能站在 Debian/nixpkgs 的人工策展之上,用 ELF 自动分析做校验补全、用 AI 做元数据翻译。
> C1 从"存亡级未知"变成"重大但有明确出路"——前提是设计承认并拥抱"继承现有生态"这条路。**
