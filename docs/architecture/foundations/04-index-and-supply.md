## 0. 设计哲学

> **Aevum 的创新在"可复现 / 原子 / AI 维护",不在"重造一个包生态"。
> 包生态站在 Debian / nixpkgs / Arch 的人工策展之上 —— 继承,而非另起炉灶。**

这是经 PoC-1 实测数据校准后的诚实定位。此前文档把"索引内容从哪来"留白,被评审列为唯一的存亡级风险(C1)。本篇正面回答它。

---

## 1. 问题:上层机制全部依赖一个未定义的输入

求解器、闭包、世代、AI 维护者,全部建立在一个前提上:

> 存在一个包索引,其中每个包都已标注 `provides`(对外能力)/ `requires`(依赖约束),且 `name@version → hash` 的绑定可信。

此前 `store.toml` 里 `origin = "aevum-index:..."` 引用了这个索引,**却从未定义它的内容从哪来**。如果答案是"自己从二进制自动生成",那要先证明这件事可行。

---

## 2. 实测裁决:纯自动不可行,但有第三条路

PoC-1 用整个 Debian stable(68,755 包、308,807 条依赖)+ 19 个真实包解 ELF,测出依赖元数据的可机器生成性:

| 类别 | 占比 | 能否自动 |
|---|---|---|
| 纯库依赖(ELF `DT_NEEDED` 可直接出) | **11.6%** | 🟢 全自动 |
| 库依赖 + 版本约束(名字自动,版本需语义) | **41.5%** | 🟡 半自动 |
| 非库 / 虚包 / alternatives(ELF 不可见) | **46.9%** | 🔴 纯人工语义 |

抽样实测:**36.8% 的人工声明依赖是 ELF 根本看不到的**(配置、daemon、数据包、运行时 dlopen)。`nginx-core` 的 14 条依赖里,ELF 能给出的 soname 数为 **0**。

**裁决**:

- 自己从零靠 ELF 生成完整可用索引 = **不可行**(近半是机器永远看不见的人工语义,且这些恰是"装完能不能跑"的关键)。
- 但分布也指出第三条路:**继承现有发行版的人工策展,用 ELF 自动分析做校验补全,用 AI 做元数据翻译**。

完整数据见 [`PoC-1 报告`](../../../poc/poc1-index-feasibility/REPORT.md)。

## 3. 供给模型:三层 bootstrap

```text
┌──────────────────────────────────────────────────────────────┐
│  上游发行版（Debian / Arch / nixpkgs …）                       │
│  人工策展的元数据:Depends / Provides / 虚包 / alternatives     │
└────────────────────────────┬─────────────────────────────────┘
                             │ ① 导入(继承那 46.9% 人工语义)
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  Aevum Index 构建管线                                          │
│  ② ELF 自动校验补全(11.6% 纯自动 + 41.5% 半自动)             │
│     - 解 DT_NEEDED 抓真实 so 依赖,交叉验证/修正继承来的元数据  │
│     - 发现上游漏标的库依赖                                      │
│  ③ AI 翻译:把上游依赖模型映射进 Aevum 能力模型(provides/req)  │
│     - 处理命名差异、能力抽象、冲突归一                          │
│  ④ 再签名:对 name@version → hash 绑定签名(信任根)            │
└────────────────────────────┬─────────────────────────────────┘
                             │ 产出
                             ▼
┌──────────────────────────────────────────────────────────────┐
│  aevum-index:已标注 provides/requires + 内容寻址 hash + 签名   │
│  → store.toml 的 origin 来源;求解器的输入快照                  │
└──────────────────────────────────────────────────────────────┘
```

### 3.1 ① 导入:继承人工策展

- 直接复用上游的 `Depends`/`Provides`/虚包/alternatives —— **那 46.9% 不自己造**。
- 上游元数据是数十年人力的结晶(nixpkgs 是 NixOS 真正的护城河),Aevum 站在它肩上。
- 导入是元数据层面的,不绑定上游的构建方式(Aevum 仍可用预编译二进制,见 [`03-binary-compat`](../runtime/03-binary-compat.md))。

### 3.2 ② ELF 自动校验补全

- 解每个包的 `DT_NEEDED`,得到它真实链接的 so(11.6% 场景能直接成依赖)。
- **角色是"校验与补全",不是"从零生成"**:用自动抽取的事实去交叉验证继承来的元数据,抓上游漏标、修正过时约束。
- 半自动那 41.5%:库名由 ELF 确认,版本约束沿用上游声明。

### 3.3 ③ AI 翻译进能力模型

- AI 维护者在索引构建链路的真正发力点:把上游的依赖表达**翻译**进 Aevum 的能力模型(`provides.capabilities` / `requires.capability`)。
- 处理:命名差异(`libssl3` ↔ `libssl.so.3` ↔ 能力 `tls`)、能力抽象、跨发行版归一、冲突。
- **不是**"让 AI 理解二进制生成依赖"(那只有 11.6% 能成);**是**"让 AI 做元数据的语义搬运工"。
- 受 ADR-0003 约束:AI 产出的映射可审计、可人工复核;关键归一决策可否决。

### 3.4 ④ 再签名(信任根)

- 对 `name@version → hash` 绑定签名,这是 [`远程缓存`](../runtime/06-remote-cache.md) §4 与评审 H1 强调的信任根。
- 内容寻址保证"字节==hash",签名保证"这个 hash 是该包该有的那个"。两层缺一不可。
- 信任根、密钥轮换/撤销、透明日志的完整设计是独立课题(评审 H1/H2),本篇只确立"索引产出必须再签名"这一接口。

---

## 4. 这如何把 C1 从"存亡级"降为"可工程化"

| 此前(被评审质疑) | 校准后 |
|---|---|
| 索引内容从哪来?未定义 | 来自上游发行版元数据导入 |
| 靠 AI/ELF 从零生成? | 否,实测只有 11.6% 纯自动;改为继承 + 校验补全 + 翻译 |
| 与 NixOS 比生态护城河? | 不与之竞争生态,**复用**其成果;竞争点在可复现/原子/AI 维护 |
| `aevum-index` 是什么? | 上游元数据导入 + ELF 校验 + AI 翻译 + 再签名 的产物 |

→ Aevum 不再是"平地起高楼建生态",而是"在成熟生态上加一层可复现 + AI 维护"。这让定位更可信,也让 C1 变成有明确路径的工程问题。

---

## 5. 诚实声明(写给作者,也写给未来用户)

- **Aevum 的包生态是借来的,不是原创的。** 创新在系统机制(内容寻址世代、AI 维护、分层隔离、二进制友好),不在"又攒了一套包"。这不丢人 —— Ubuntu 借 Debian、众多发行版借上游,都是站在巨人肩上。
- 继承上游元数据带来依赖:上游元数据的质量、许可、更新节奏会影响 Aevum。需评估各上游的许可是否允许再分发元数据。
- AI 翻译这一环若出错会污染索引,因此它**不在运行时热路径**,而在离线的索引构建管线 + 人工复核 + 签名之后才进入用户可见的 index。
- 冷启动现实:初期可只覆盖一个上游(如 Debian)的子集,逐步扩大;不必一开始就全量。

## 6. 与冷启动战略的关系

本篇定义"索引怎么来",[`../../comparison/01-nixos-pain-points.md`](../../comparison/01-nixos-pain-points.md) §7 的冷启动战略章节定义"生态怎么从小长大"。两者配合:技术上能继承(本篇)+ 策略上分阶段铺开(痛点文 §7)。

## 7. 验收清单

- [ ] 索引构建管线四步(导入/校验/翻译/签名)成形
- [ ] 导入:复用至少一个上游(Debian)的依赖元数据
- [ ] ELF 校验:解 DT_NEEDED,交叉验证继承元数据,报告差异
- [ ] AI 翻译:上游依赖 → Aevum 能力模型,可审计可复核
- [ ] 再签名:name@version→hash 绑定签名,客户端验签
- [ ] AI 翻译不在运行时热路径(离线管线 + 复核 + 签名后才生效)
- [ ] 上游元数据许可合规评估
- [ ] store.toml 的 origin 指向本管线产出
