# 状态 vs 包回滚 —— 可变数据的协调回退

## 0. 设计哲学

> **世代回滚只切回"装了什么";但"数据写成了什么"不会自己跟着回去。
> 包退了、数据没退 = 炸。Aevum 必须把这两者协调起来,而不是假装数据不存在。**

这是连 NixOS 都没解好的硬骨头,也是 Aevum 此前文档的真空。补上它是差异化。

---

## 1. 问题:两种"状态"是两回事

```text
┌─────────────────────────────┬──────────────────────────────┐
│  包/配置状态（不可变)        │  可变运行状态（mutable)        │
├─────────────────────────────┼──────────────────────────────┤
│  装了哪些包、什么版本、配置  │  数据库内容、用户文件、缓存    │
│  内容寻址、世代化            │  随程序运行不断被写            │
│  回滚 = 切 active 指针(秒级) │  回滚 = ？(世代机制管不到)     │
└─────────────────────────────┴──────────────────────────────┘
```

**危险场景**:

```text
gen-N:   postgresql@16，数据目录已被 16 升级成新的磁盘格式
   ↓ 用户回滚
gen-(N-1): postgresql@15
   ↓
postgres 15 的二进制去读 16 写的数据目录
   → 格式不认 → 启动失败 / 数据损坏
```

世代回滚把**包**退回去了,但磁盘上的**数据**还是新格式。包和数据脱节,直接出事。

---

## 2. 状态的三种类别(决定怎么处理)

不是所有可变状态都要回退。先分类:

| 类别 | 例子 | 回滚策略 |
|---|---|---|
| **关键有状态数据** | 数据库数据目录、关键服务的持久卷 | **快照协调**:随世代一起做快照,一起回退 |
| **用户数据** | `~/` 下的文档、项目文件 | **默认不随世代回退**(用户的东西归用户),但提供独立快照线 |
| **可重建/瞬态** | 缓存、日志、临时文件、`/tmp` | **不回退**,甚至回滚时可清掉 |

> 核心判断:**回退一个状态,是让系统更一致还是更混乱?** 数据库数据不回退 → 和退回的包不匹配(更乱);用户文档回退 → 用户丢了刚写的东西(更乱)。所以前者随包退,后者不随。

## 3. 机制:状态快照与世代绑定

### 3.1 借力文件系统子卷快照

Aevum 不自己重造快照,借底层 FS(btrfs / zfs)的子卷(subvolume)快照能力 —— 这是 Snapper 的成熟路子:

```text
关键有状态数据放在独立子卷：
  /aevum/state/postgresql/   ← btrfs subvolume

激活新世代(涉及有状态服务变更)前：
  1. 对相关 state 子卷做快照 state-snap@gen-N
  2. 把快照 id 记进 gen-N 的 manifest（状态绑定)
  3. 再 activate 新世代

回滚到 gen-(N-1) 时：
  1. 切 active 指针(包/配置回退) ── 已有机制
  2. 读 gen-(N-1) manifest 里绑定的 state-snap@gen-(N-1)
  3. 把对应 state 子卷回滚到那个快照 ── 新增
  → 包和数据一起回到 gen-(N-1) 的一致点
```

### 3.2 世代 manifest 扩展

在 [`../foundations/02-generation.md`](../foundations/02-generation.md) 的 manifest 里增加状态绑定:

```toml
# generations/gen-N/manifest.toml （扩展）
[[state_binding]]
service = "postgresql"
subvolume = "/aevum/state/postgresql"
snapshot_id = "state-snap-pg-gen-N"      # 激活前打的快照
fs = "btrfs"
policy = "snapshot-with-generation"      # 见 §2 分类

[[state_binding]]
service = "user-home"
subvolume = "/home"
policy = "independent"                    # 不随世代回退,独立快照线
```

## 4. 回滚时的协调流程

```text
用户/AI 发起回滚 gen-N → gen-(N-1)
        ↓
1. 读 gen-(N-1) manifest 的 state_binding
        ↓
2. 对每个 binding 按 policy 处理：
   ├─ snapshot-with-generation → 回滚该 state 子卷到绑定快照
   ├─ independent              → 不动(给用户单独的快照管理界面)
   └─ ephemeral                → 不动 / 可清理
        ↓
3. 切 active 指针(包/配置回退)
        ↓
4. 协调校验：退回的包与退回的状态是否一致
   （如 postgres 15 + 15 时代的数据快照 → 匹配)
        ↓
5. 启动 → 一致点恢复
```

**关键**:状态回退和指针切换要**作为一个事务**协调,任一步失败则整体中止,避免"包退了状态没退"的半回滚态。

---

## 5. 不可逆操作的护栏

有些状态变更天然不可逆,快照也救不全(如已经发出去的邮件、已扣款的交易)。Aevum 的态度是**诚实标注 + 激活前警告**:

```text
AI 在 diff/plan 阶段(见 runtime/05-generation-diff-plan.md)标注：
  ⚠ 此变更涉及 postgresql 大版本升级(16→17)
    数据目录将被迁移,迁移后回滚需要靠 state 快照
    建议:激活前确认快照已就绪
  ⚠ 此操作涉及外部副作用(发送通知),不可通过回滚撤销
```

→ 不假装"什么都能回滚",而是让用户在不可逆操作前知情。

---

## 6. 与 impermanence 思路的呼应

NixOS 的 impermanence 实践:把系统设计成"每次启动状态从已知基线重建,只显式保留该保留的"。Aevum 可借此思路做可选模式:

```text
可选的"清爽模式"：
  - App 层运行状态默认瞬态,启动从世代重建
  - 只有显式声明为持久的 state 子卷才保留
  → 减少状态漂移,让"系统 = 世代"更接近真实
```

这是可选策略,不强制 —— 桌面用户可能更想要"东西都留着"。

---

## 7. 边界与诚实声明

- Aevum **不**重造快照引擎,依赖底层 FS(btrfs/zfs)能力;不支持这类 FS 的环境,状态回滚降级为"仅包回滚 + 明确警告"。
- 状态回滚的事务性、跨多个有状态服务的一致性,是实现期的硬课题,本文定语义契约,细节待代码阶段。
- 用户数据默认不随世代回退,是有意的安全选择(宁可不回退用户文档,也不能让用户丢数据)。
