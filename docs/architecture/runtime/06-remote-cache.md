## 0. 设计哲学

> **内容寻址的红利:hash 一样,内容就一样,谁先有了大家都能用。
> 不重复下载、不重复构建、只传缺的那部分。**

store 是内容寻址的(见 [`../foundations/01-store.md`](../foundations/01-store.md)),这天然支持"远程缓存":一个 hash 在哪台机器上都是同一份内容,可以从最近的来源取,不必每台机器各自构建/下载全量。

---

## 1. 两个问题,两个机制

| 问题 | 机制 | 对标 |
|---|---|---|
| 别人已经构建/拥有的包,我为何要重下/重建? | **远程缓存(substituters)** | Nix binary cache |
| 同步一个世代,为何要传全量而非只传"我缺的"? | **增量传输(去重拉取)** | OSTree |

## 2. 远程缓存(Substituters)

### 2.1 模型

```text
求解出 closure 需要 {hash-A, hash-B, hash-C}
        ↓
对每个 hash,按优先级找来源:
  1. 本地 store         已有 → 0 成本(去重)
  2. 局域网缓存/镜像     有 → 局域网速度拉取
  3. 远程二进制缓存      有 → 下载预构建产物(免本地构建)
  4. 从源构建/官方源     都没有 → 兜底
        ↓
拿到后校验 hash(内容寻址保证拿对了)→ 入 store
```

### 2.2 配置(草案)

```toml
# 远程缓存来源,按优先级
[[substituter]]
url = "https://cache.aevum.example"
priority = 10
trusted_keys = ["aevum-cache-2026:..."]   # 只信任签名的产物

[[substituter]]
url = "http://lan-mirror.local:8080"       # 局域网镜像,更快
priority = 5

[cache]
verify_signature = true                     # 强制验签(见 §4)
fallback_to_build = true                    # 缓存没有时允许本地构建/官方源
```

### 2.3 为什么内容寻址让这安全

```text
从任何来源(哪怕不可信的镜像)下载 hash-A 的内容
        ↓
本地重算内容哈希,必须 == "A"
        ↓
不等 → 丢弃(被篡改/损坏),换来源
等 → 内容必然就是我要的那一份(hash 即身份)
```

→ 缓存来源**不需要被信任内容正确性**,内容寻址本身就是防篡改校验。签名(§4)是额外一层(防"投毒一个假 hash 进索引")。

## 3. 增量传输(去重拉取)

### 3.1 模型

```text
同步世代 gen-8 到一台新机器(或推送到服务器):
  不传 gen-8 的全部 145 个包
        ↓
  对比双方已有的 hash 集合
        ↓
  只传"目标缺的那些 hash"(差集)
        ↓
  常见情况:两个世代 90% 包相同,只传那 10% 的差异
```

这正是 OSTree "git for OS" 的思路:对象级去重 + 只传增量。对 Aevum:

- 世代间高度共享 hash(升级往往只动几个包),增量传输让"切世代/同步世代"极省带宽。
- 内容寻址让"对方有没有这个 hash"是一个精确的集合运算,不需要逐文件比对。

### 3.2 块级去重(可选深化)

```text
更细一层:同一个大包的两个版本,内部很多文件/块相同
        ↓
块级内容寻址(类似 OSTree/casync 的 chunking)
        ↓
传输时连"包内相同的块"都能跳过 → 进一步省带宽
```

这是可选优化,语义上对上层透明(上层只看到 hash 级)。

## 4. 安全:签名与信任

```text
两层防护:
  1. 内容寻址 → 防"下载的内容和 hash 不符"(传输损坏/篡改)
  2. 签名     → 防"索引里登记了一个恶意 hash"(供应链投毒)

签名链:
  可信构建方对 (包名@版本 → hash) 的映射签名
  客户端用 trusted_keys 验签
  → 确保"我要的 python@3.11 对应的 hash"是可信方背书的,
     而不是攻击者塞进索引的恶意 hash
```

- `verify_signature = true` 时,未签名/验签失败的产物拒绝使用。
- 这与 Foundation 的签名机制([`../../layers/01-foundation.md`](../../layers/01-foundation.md) §2)是同一套信任理念,延伸到普通包缓存。

## 5. 与调和循环、AI 的协作

```text
调和循环(ai/04)发现索引有安全更新
        ↓
评估更新涉及的 hash 是否已在远程缓存有预构建产物
        ↓
有 → 提议的世代可快速拉取(免构建),plan 里标注"已缓存,下载约 N MB"
没有 → plan 里标注"需本地构建,预计耗时 M 分钟"
        ↓
让用户在 plan 阶段(runtime/05)对"代价"知情
```

→ 远程缓存的"有没有现成产物"成为 plan 的一个输入,让 AI 能预告"这次更新快不快"。

---

## 6. 离线与弱网

```text
- 离线:只用本地 store;缺的 hash → 明确报"需联网获取 X"
- 弱网:优先局域网镜像;支持断点续传(内容寻址下,已传的块不重传)
- 完全离线分发:可导出一个世代的完整 closure 为离线包,
  在隔离环境导入(对标 nix copy / 离线 bundle)
```

---

## 7. 边界与诚实声明

- 远程缓存是**性能优化**,不改变可复现语义:有没有缓存,最终得到的 hash 完全一样,只是快慢之分。
- 缓存来源可不可信都行(内容寻址兜底),但**索引映射**必须经签名(防投毒)。
- 块级去重、casync 式 chunking 是可选深化,先做 hash 级足够;细节待实现期。
