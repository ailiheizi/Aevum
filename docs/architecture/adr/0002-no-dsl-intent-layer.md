# ADR-0002:意图层不引入图灵完备 DSL

> 状态:**已接受(Accepted)** · 日期:2026-06-08
> 父文档:[`../00-overview.md`](../00-overview.md)
> 关联:[`../runtime/02-intent-resolved-lock.md`](../runtime/02-intent-resolved-lock.md)、[`../../comparison/01-nixos-pain-points.md`](../../comparison/01-nixos-pain-points.md)、模板 [`../../templates/`](../../templates/)

---

## 背景

NixOS 用 Nix 语言(图灵完备、惰性求值、纯函数式)描述系统。这是它可复现的来源,**也是它最大的劝退点**(见痛点文档 §1):学习曲线陡、报错难读、知识不可迁移。

Aevum 要"比 NixOS 智能且更易用"。一个绕不开的根本抉择:**用户表达系统配置时,要不要一门编程语言?**

---

## 决策

**Aevum 的意图层(用户面对的配置)是纯声明式数据,不引入任何图灵完备 DSL。**

- 意图用 TOML 这类纯数据格式表达:模板引用 + 声明式覆盖 + 排除。
- 一切需要"逻辑"的地方(条件、组合、依赖求解、版本选择)由**模板系统 + AI 维护者 + 确定性求解器**承担,而不是让用户写表达式。

---

## 理由

### 直接消灭 NixOS 最大的墙

DSL 是 NixOS 学习曲线、报错地狱、知识不可迁移的总根源。不引入 DSL = 从源头消除这一整类痛点。会读 TOML 的人就能看懂、改动 Aevum 配置。

### 逻辑的位置变了,而非消失了

系统配置确实需要逻辑(条件启用、组合、解依赖)。NixOS 把这份逻辑**推给用户**(你写 Nix)。Aevum 把它**收进系统**:

```text
NixOS:   用户写逻辑（Nix 表达式） → 求值 → 系统
Aevum:   用户写意图（数据）→ 模板提供组合 → AI 翻译 → 求解器算 → 系统
```

逻辑没消失,而是从"用户负担"变成"系统能力"。这正是"更智能"的含义。

### 不牺牲可复现

有人会担心:没有 DSL 还能可复现吗?能。可复现的真正来源是 **lock 锁定**(内容寻址 + 索引快照 + 闭包),不是 DSL 本身。NixOS 的可复现也主要靠 store + lock,Nix 语言只是描述手段。Aevum 用数据 + 模板 + 求解器描述,lock 一样锁死结果(见 [`../runtime/02-intent-resolved-lock.md`](../runtime/02-intent-resolved-lock.md))。

### 保留可审计与可重放

- AI 的求解过程落在 resolved.toml(可审计、可否决)。
- 最终结果落在 lock.toml(可重放,无需 AI、无需 DSL 求值)。

---

## 代价与应对

| 代价 | 应对 |
|---|---|
| 极端定制场景,纯数据可能不如代码灵活 | 用派生模板 + override 覆盖绝大多数需求;真正罕见的极端需求由 AI 维护者协助,而非逼所有人学 DSL |
| 失去 Nix 那种"配置即程序"的强表达 | 这正是目标 —— 表达力换易用性,对桌面/开发者主场景是正确取舍 |
| 模板系统需要足够强 | 投入设计模板的组合/派生/覆盖语义(见 [`../../templates/01-template-model.md`](../../templates/01-template-model.md)) |

---

## 边界

- 意图层(intent.toml)**严禁**任何可执行逻辑(无函数、无条件表达式、无循环)。
- 模板(*.toml)是声明式蓝图,也不是脚本;组合靠 `extends` 等声明字段,不靠代码。
- 需要图灵完备计算的地方,只能存在于 AI 维护者和求解器的实现内部,**永不暴露给用户配置**。

---

## 影响

- 这是 Aevum 区别于 NixOS/Guix 的**身份级决策**,不可在后续随意推翻(推翻 = 变回 NixOS)。
- 所有面向用户的配置文档以"纯数据 + 模板"为准。
- 模板系统的强弱直接决定本决策能否成立,是设计重点。
