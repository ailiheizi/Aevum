# ADR-0001:定位为对标 NixOS 的 AI-native 用户态系统层

> 状态:**已接受(Accepted)** · 日期:2026-06-08
> 父文档:[`../00-overview.md`](../00-overview.md)
> 关联:[`../../comparison/02-prior-art.md`](../../comparison/02-prior-art.md)

---

## 背景

Aevum 要做"AI-native 的 NixOS 替代"。开工前必须钉死一个根本边界:Aevum 到底是什么层级的东西?这决定所有后续文档怎么写、实现从哪里切入。

候选定位有三:

1. **完整独立发行版**:从 bootloader/内核起全是自己的。
2. **用户态系统层**:建在现有 Linux 内核 + 文件系统之上,自己管"AI 维护的可复现软件世代"。
3. **跨平台环境/包管理器**:不绑定 OS,在任意系统上管软件栈。

---

## 决策

**Aevum 定位为:对标 NixOS 的、AI-native 的用户态系统层 —— 建立在 Linux 内核与文件系统之上,主打桌面、兼顾服务器。**

- 对用户而言,它是"我的操作系统环境"(像 NixOS 一样管理整个系统的软件、服务、依赖)。
- 对实现而言,它**复用宿主 Linux 的内核与驱动**,自己专注于内容寻址 store、世代、分层、AI 维护链路。
- 不在第一阶段啃 bootloader/内核/驱动这些深渊。

---

## 理由

### 为什么不是完整独立发行版(候选 1)

- 内核、驱动、硬件适配、init 生态是巨大的工程深渊,且**不是 Aevum 的差异化所在**。Aevum 的价值在"AI 维护的可复现世代",不在重写内核。
- NixOS 本身也是建在 Linux 内核上的发行版,对标它不需要自己造内核。
- 可演进:用户态系统层成熟后,未来要做成可引导发行版(打包内核 + Aevum 用户态)是增量工作,不必一开始就背。

### 为什么不是纯跨平台包管理器(候选 3)

- 用户明确要的是"NixOS 替代",是**系统**级的可复现与原子化,不是又一个 `brew`/`conda`。
- 跨平台会迫使放弃很多系统级保证(分层隔离、foundation-only 兜底、二进制兼容都强依赖类 Linux 环境)。
- 失去"整个系统一个世代"的心智,就失去了 NixOS 最珍贵的东西。

### 为什么是用户态系统层(候选 2,选中)

- 保住 NixOS 的全部核心价值(系统级可复现、原子世代、回滚),同时把内核深渊留给宿主。
- 务实、能快出原型、风险可控。
- 与竞品 osModa 形成清晰差异:osModa 建在 NixOS 之上保留 Nix 语言;Aevum 在 Linux 之上**替换**意图层。

---

## 边界(本决策明确划定的范围)

| 在 Aevum 范围内 | 不在范围内(交给宿主 Linux / 未来) |
|---|---|
| 内容寻址 store | Linux 内核 |
| 世代管理与回滚 | 硬件驱动 |
| Foundation/System/App 分层 | bootloader(第一阶段) |
| AI 维护者与依赖求解 | 内核态模块 |
| 二进制兼容(用户态链接器/基线) | 内核 ABI |
| 模板系统 | —— |

---

## 影响

- 所有文档以"用户态系统层"为前提书写。
- 二进制兼容文档([`../runtime/03-binary-compat.md`](../runtime/03-binary-compat.md))聚焦用户态链接器与共享库基线,不碰内核。
- Foundation 层([`../../layers/01-foundation.md`](../../layers/01-foundation.md))的 `init` 指 Aevum 用户态的初始化主体,不取代宿主内核 init(具体协作关系待实现阶段定)。

---

## 后续可重新审视的条件

- 若用户态层成熟且有强需求,可启动"可引导发行版"增量项目(打包内核)。
- 若出现必须深入内核态才能解决的可复现/隔离问题,再单独立 ADR 讨论。
