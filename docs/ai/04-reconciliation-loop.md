## 0. 设计哲学

> **第一维护者不是"等你下命令的命令行",而是"持续盯着系统、发现偏差就收敛回去的那个人"。
> 声明期望态,让系统不断向它收敛 —— 这才配叫"AI 维护"。**

[`01-maintainer-loop.md`](01-maintainer-loop.md) 描述的是"响应一次意图"的循环。本篇把它升级为**持续运行的调和循环**:这是 GitOps(ArgoCD/Flux/K8s controller)的内核,与"AI 是第一维护者"的定位天然契合。

---

## 1. 命令式 vs 声明式收敛

```text
命令式(传统包管理器 apt/dnf):
  你喊 "装 X" → 执行一次 → 完事
  之后 X 被改坏、配置漂移、依赖被别的操作破坏 → 没人管,直到你再发现

声明式调和(Aevum Maintainer):
  你声明 "期望态 = gen-8" → 循环:
     读实际态 → 和期望态 diff → 有偏差就提议/执行收敛 → 再读 …
  系统永远朝期望态收敛,漂移被主动发现
```

期望态 = 当前 active 世代(那份 lock 就是"声明的真理")。实际态 = 磁盘上 store/链接/运行服务的真实状况。**调和 = 让实际态持续等于期望态。**

## 2. 调和循环

```text
        ┌──────────────────────────────────────────────┐
        │                  调和循环(常驻)               │
        └──────────────────────┬───────────────────────┘
                               ▼
                   ┌───────────────────────┐
                   │ 1. 观测实际态          │
                   │  store hash 完整性     │
                   │  链接树是否符合 lock   │
                   │  关键服务是否在跑      │
                   │  索引是否有安全更新    │
                   └───────────┬───────────┘
                               ▼
                   ┌───────────────────────┐
                   │ 2. 与期望态 diff       │  期望 = active 世代的 lock
                   └───────────┬───────────┘
                          ┌────┴────┐
                     一致 │         │ 偏差
                          ▼         ▼
                   ┌──────────┐  ┌─────────────────────────┐
                   │ 休眠到   │  │ 3. 分类偏差              │
                   │ 下个周期 │  │   可自动修正 vs 需决策   │
                   └──────────┘  └────────────┬────────────┘
                                         ┌────┴────┐
                              自愈(安全) │         │ 需人决策
                                         ▼         ▼
                              ┌──────────────┐  ┌──────────────┐
                              │ 4a. 自动收敛  │  │ 4b. 提议+plan │
                              │  重拉损坏hash │  │  推询问列表    │
                              │  修复链接树   │  │  等确认        │
                              └──────────────┘  └──────────────┘
```

## 3. 偏差分类(决定自动还是问人)

| 偏差类型 | 例子 | 处理 |
|---|---|---|
| **完整性漂移** | store 某 hash 损坏、链接丢失 | **自愈**:按 lock 重拉/重建(安全方向,期望态本就该是这样) |
| **服务漂移** | 期望在跑的服务挂了 | **自愈**:按世代定义重启;反复失败 → 升级为需决策 |
| **可用更新** | 索引出现安全补丁 | **提议**:propose 候选世代 → plan → 问人(改变期望态,需授权) |
| **配置漂移** | 实际配置被外部改动偏离 lock | 默认**告警 + 提议收敛**(避免和用户手动改动打架,见 §5) |
| **状态异常** | 有状态服务数据异常 | **不擅自动**:涉及数据,提议 + 警告(见 [`../architecture/runtime/04-state-vs-package-rollback.md`](../architecture/runtime/04-state-vs-package-rollback.md)) |

> 原则:**朝期望态收敛且无代价无风险 → 自愈;改变期望态或涉及数据/代价 → 提议 + plan + 问人。** 这直接复用 ADR-0003 的三边界。

## 4. 自愈 vs 改变期望态(关键区分)

调和循环有两种动作,千万别混:

```text
自愈(reconcile to desired):
  实际态偏离了期望态 → 把实际态拉回期望态
  例:lock 说该有 hash-X,但磁盘上损坏了 → 重拉 hash-X
  这不改变 active 世代,只是恢复它本该的样子 → 安全,可自动

改变期望态(change desired):
  期望态本身要变(装新软件、升级、应用安全补丁)
  → 必须走 propose→verify→plan→activate,必须问人
  这是 01-maintainer-loop.md 描述的那条路
```

调和循环新增的价值在**自愈**那一半:系统会自己保持健康,而不是等你发现它坏了。

## 5. 与用户手动改动的边界(防止"AI 跟你抢")

声明式系统的经典摩擦:用户手动改了点东西,调和循环把它"纠正"回去,用户火大。Aevum 的态度:

```text
1. App/用户层的手动改动 → 默认尊重,不强行覆盖;
   检测到漂移 → 告警 + 询问"要把这个改动固化进世代吗?"
   （把临时改动"提升"为声明的一部分,而非粗暴抹掉)
2. Foundation/System 关键完整性漂移(损坏)→ 自愈(这是损坏不是用户意图)
3. 永不静默覆盖用户的有意改动 → 区分"损坏"与"用户故意改的"
```

→ 调和不是独裁。它收敛"坏掉的",尊重"故意的",把"故意的"引导成声明。

## 6. 节奏与触发

```text
- 周期性 tick(如每数分钟)做轻量完整性/服务巡检
- 事件驱动:服务崩溃、磁盘事件、索引更新通知 → 立即调和
- 远程缓存/索引上游变化 → 触发"有更新"评估(见 ../architecture/runtime/06-remote-cache.md)
- 退避:反复失败的收敛不无限重试,升级为需人决策(避免 crashloop)
```

## 7. 和现有文档的关系

调和循环不是新机制,是把已有部件串成"持续"形态:

```text
观测      → 复用 store 完整性校验(foundations/01)
diff      → 复用世代 diff(runtime/05)
收敛(改) → 复用 propose→verify→plan→activate(runtime/01,05)
自愈      → 按 active lock 重建(本篇新增的"持续"视角)
边界      → 复用 ADR-0003 三边界
回滚      → 复用 rollback(runtime/01)
```

本篇的增量是**"常驻 + 主动 + 自愈"**这层语义,把 Maintainer 从"命令响应器"正式定义为"自治收敛器"。

---

## 8. 边界

1. 自愈只朝 active 世代的 lock 收敛,**不擅自改变期望态**。
2. 改变期望态一律走 plan + 问人(ADR-0003)。
3. 涉及可变数据的偏差不擅自动(runtime/04)。
4. 尊重用户有意改动,区分"损坏"与"故意",不独裁覆盖。
5. 失败收敛有退避,不 crashloop。

- [ ] 改变期望态走 plan + 问人
